Data Lineage Platform Architecture

How components interact: ai-infra → Backend → Frontend

System Architecture
🐳 ai-infra (Docker Compose)
├── MSSQL Server (localhost:1433)
├── Neo4j (localhost:7687)
└── Other services
⚙️ lineage-backend (Python FastAPI)
├── .env (points to ai-infra services)
│ ├── MSSQL_HOST=localhost:1433
│ └── NEO4J_URI=bolt://localhost:7687
├── app/core/config.py (Pydantic Settings)
├── db/mssql.py (SQLAlchemy connection)
├── db/neo4j.py (Neo4j driver connection)
├── api/routes/ (40+ endpoints)
└── services/ (business logic)
⚛️ lineage-frontend (React + TypeScript)
├── .env (VITE_API_BASE_URL=http://localhost:8000/api)
├── services/apiClient.ts (REST client)
├── features/ (screens)
└── components/ (shared UI)
Data Flow During Development
Phase 1: Setup (Day 1)
1. Clone monorepo
2. Run: ai-infra/docker-compose up
3. Backend connects to running MSSQL + Neo4j
4. Run seed scripts to populate databases
Phase 2: Build In Parallel
Track A: Backend APIs
→ Implement routes that call MSSQL/Neo4j
→ Test with curl / Swagger UI
Track B: Frontend UI
→ Build components with mock API responses
→ Use hardcoded data initially
Track C: Database Setup
→ Fine-tune schemas + indexes
→ Load realistic seed data
Phase 3: Integration (Week 2+)
React UI → REST API → MSSQL / Neo4j
Frontend switches from mock to real API calls
End-to-end testing with real data

Key Points

  • Separation of Concerns: ai-infra owns infrastructure, lineage-platform owns app logic
  • Configuration: Each component reads its own .env, references ai-infra services by hostname/port
  • Independent Work: Backend, frontend, and DB setup can progress in parallel
  • Local Development: Single docker-compose.yml in ai-infra spins up all services
